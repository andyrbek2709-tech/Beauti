# Универсальная онлайн-запись: фазы реализации (Telegram + Web + Railway)

## 1) Цель MVP
- Единая система записи для многих салонов (multi-tenant).
- Клиент может записаться через Telegram или через мобильную веб-версию.
- Все слоты синхронизированы автоматически между каналами.
- Админ (парикмахер) управляет расписанием через Telegram или веб-кабинет.
- Мобильная веб-версия должна быть не только функциональной, но и визуально сильной (удобной и красивой).

## 2) Базовые инварианты системы
- Один backend API и одна PostgreSQL БД как единый источник истины.
- Любая запись/отмена в одном канале сразу отражается в другом.
- Один слот времени может быть занят только одним клиентом.
- Все операции бронирования проходят через единый транзакционный серверный поток.

## 3) Архитектура (Railway-ready)
- `booking-api`: HTTP API + Telegram webhook.
- `booking-worker`: напоминания и фоновые задачи.
- `postgres`: хранилище расписания, слотов, записей, аудита.
- Telegram клиент и Web клиент используют одни и те же endpoint.
- Telegram админ и Web админ используют единые admin endpoint.
- Подключение салонов через invite-flow (owner создает ссылку, салон активируется self-service).

## 4) Фаза 0 — фиксация бизнес-правил
- Зафиксировать длительность записи: `30 | 45 | 60` минут.
- Зафиксировать коридоры/диапазоны записи по дням недели.
- Зафиксировать горизонт записи (по умолчанию 14 дней).
- Зафиксировать cutoff отмены (например 2-4 часа).
- Зафиксировать timezone мастера и хранение времени в UTC.

Результат фазы:
- Подтвержденный набор правил, по которым считается доступность и принимается бронь.

## 5) Фаза 0.5 — UX/UI дизайн-подготовка (обязательная)
- Зафиксировать UX-потоки мобильной версии:
  - выбор даты;
  - выбор времени;
  - подтверждение записи;
  - успешная запись;
  - конфликт слота;
  - отмена/перенос.
- Подготовить UI-kit и дизайн-токены:
  - палитра, типографика, отступы, радиусы;
  - состояния кнопок и полей (`default/disabled/error/loading`);
  - компоненты (`SlotCard`, `CalendarDay`, `PrimaryButton`, `Toast`).
- Согласовать mobile-first критерии:
  - tap targets не меньше 44px;
  - минимум 2-3 шага до подтверждения;
  - высокая контрастность и читаемость.
- Подготовить макеты для:
  - клиентского web-потока;
  - web-кабинета админа (логин, Telegram токен+ID, график).

Результат фазы:
- Утвержденные UX-сценарии и дизайн-система MVP, по которым фронт реализуется без импровизации.
- Артефакты фазы:
  - `docs/design/UX_UI_BRIEF.md`
  - `docs/design/SCREEN_SPECS.md`
  - `docs/design/DESIGN_TOKENS.json`

## 6) Фаза 1 — доменная модель и БД
- Таблицы:
  - `master_settings`
  - `working_rules`
  - `schedule_exceptions`
  - `appointments`
  - `idempotency_keys`
  - `audit_logs`
- Ограничения:
  - уникальность активного слота (`master_id + start_at`, `status='booked'`);
  - индексы для быстрых выборок по дате и мастеру.
- Подготовить миграции и стартовые данные.

Результат фазы:
- Двойная запись в один слот невозможна на уровне БД.

## 7) Фаза 2 — ядро API бронирования
- Публичные endpoint:
  - `GET /availability?from&to`
  - `POST /book`
  - `POST /cancel`
  - `GET /booking/:id`
- Админ endpoint:
  - `PUT /admin/settings`
  - `PUT /admin/working-rules`
  - `PUT /admin/exceptions`
  - `GET /admin/appointments?date`
- В `POST /book`:
  - финальная recheck-проверка слота в транзакции;
  - запись брони;
  - при конфликте возврат `409 slot_unavailable`.
- Добавить идемпотентность по `request_id`.

Результат фазы:
- Рабочее и защищенное от гонок API для Telegram и Web.

## 8) Фаза 3 — автоматическая синхронизация каналов
- `availability` вычисляется только из актуальных данных БД.
- Перед подтверждением записи клиент делает повторную валидацию слота.
- При отмене слот автоматически снова попадает в доступные.
- Для MVP синхронизация через polling/recheck перед подтверждением.

Результат фазы:
- Если слот заняли в Telegram, он недоступен в Web, и наоборот.

## 9) Фаза 4 — Telegram каналы (клиент + админ)
- Клиентский flow:
  - выбрать дату -> выбрать время -> подтвердить -> получить подтверждение.
- Отмена записи с учетом cutoff-ограничений.
- Админский flow:
  - настройка коридоров записи;
  - настройка длительности `30/45/60`;
  - исключения по датам;
  - просмотр записей за день;
  - уведомления о новых/отмененных записях.
- Базовый живой диалог в Telegram:
  - команды админа: `/help`, `/status`, `/today`, `/link`;
  - клиентский автоответ с ссылкой на web-запись.

Результат фазы:
- Полноценная работа через Telegram как для клиентов, так и для мастера.

## 10) Фаза 5 — мобильная Web-версия (Android/iPhone)
- Реализация строго по утвержденным UX/UI макетам и дизайн-токенам из Фазы 0.5.
- Mobile-first интерфейс записи (2-3 шага без перегрузки).
- Совместимость с современными мобильными браузерами:
  - Android Chrome
  - iPhone Safari/Chrome
- Fallback-поведение для устаревших браузеров (упрощенный UI и polling).
- Веб-кабинет админа:
  - авторизация под аккаунтом мастера;
  - управление графиком и длительностью слота;
  - просмотр записей по дню.

Результат фазы:
- Единый пользовательский опыт в мобильной веб-версии и общий админ-контур.

## 11) Фаза 6 — Railway deployment и эксплуатация
- Развернуть сервисы `booking-api`, `booking-worker`, `postgres`.
- Настроить переменные:
  - `DATABASE_URL`
  - `BOT_TOKEN`
  - `MASTER_TELEGRAM_ID`
  - `BOOKING_HORIZON_DAYS`
  - `CANCEL_CUTOFF_HOURS`
  - `TZ`
- Настроить Telegram webhook на Railway URL.
- Добавить owner-панель для генерации invite-ссылок без ручных API-запросов.
- Добавить управление жизненным циклом инвайтов (active/used/expired/revoked).
- Добавить фильтрацию и быстрый поиск инвайтов в owner-панели.
- Добавить billing stub с trial (14 дней) и owner-метрики по салонам/инвайтам/записям.
- Добавить Telegram webhook security (`secret token`) и dedup по `update_id`.
- Добавить автонастройку Telegram-бота и быстрый health-check подключения в админке.
- Добавить встроенную user-friendly инструкцию по созданию Telegram-бота для салонов.
- Добавить быстрый шаблон обращения в поддержку для Telegram-интеграции.
- Добавить one-click проверку готовности Telegram-подключения с понятным итогом.
- Добавить базовые технические метрики и панель статуса системы для owner.
- Отшлифовать first-run UX по инвайту: одна форма активации без дублирующих шагов.
- После first-run активации переводить пользователя в обычный вход, сохраняя auto-setup Telegram и уведомления о примененных настройках.
- Включить:
  - health endpoint;
  - структурные логи;
  - резервные копии БД;
  - проверку восстановления backup.

Результат фазы:
- Готовый прод-контур на Railway.

## 12) Фаза 7 — тесты и go-live
- Интеграционные тесты:
  - две параллельные записи в один слот -> только одна успешна;
  - повтор `POST /book` с тем же `request_id` -> без дубля;
  - отмена освобождает слот;
  - изменение графика сразу влияет на `availability`.
- Smoke-тесты после деплоя.
- Подготовить runbook инцидентов (API/worker/DB/webhook).
- UX acceptance тесты:
  - подтверждение записи на мобильном за <= 60 секунд;
  - понятные ошибки на конфликте слота;
  - визуальная консистентность экранов.
- Автоматизированная проверка гонок:
  - `npm run test:e2e:race` (много одновременных попыток на один слот).
- Автоматизированная проверка идемпотентности:
  - `npm run test:e2e:idempotency` (повтор одного запроса не создает дубль).
- Автоматизированная проверка отмены:
  - `npm run test:e2e:cancel-release` (после отмены слот снова доступен).

Результат фазы:
- Стабильный MVP, готовый к реальному использованию.

## 13) Критерии готовности
- Нельзя создать две активные записи на один слот.
- Telegram и Web показывают одинаковую доступность.
- Админ может менять график и длительность `30/45/60` через Telegram и Web.
- Любые изменения графика сразу отражаются в выдаче слотов.
- Уведомления мастеру приходят стабильно.
- Ключевые интеграционные тесты проходят.
- Мобильный UI принят по дизайн-чеклисту (удобство, читаемость, визуальная целостность).
